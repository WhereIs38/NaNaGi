# NaNaGi 关联检索监督机制 — 技术论证

> 2026-06-16
> 论证对象：[[关联]] 图遍历的隧道视野防御机制
> 核心问题：如何在不依赖粗糙跳数限制的前提下，智能检测并终止过深检索

---

## 1. 问题定义

### 1.1 隧道视野的精确描述

```
用户问: "GNN-B 的朋友圈权重方案后来怎么样了？"

MEMORY.md 命中 → GNN-B边权重设计.md
  └─ [[关联]] → gnn-dien-temporal-interest.md (GNN+DIEN时序)
       └─ [[关联]] → sparrl-positioning-decision.md (SparRL定位决策)
            └─ [[关联]] → nanagi-psychology-research-device.md (心理学研究装置)
                 └─ 🔴 已完全偏离GNN话题，进入NaNaGi设计哲学

跳数: 4。第3-4条记忆与原始问题无关，但被图边牵引至此。
```

根因：**图边编码的是"曾经一起讨论过"的共现关系，而非"针对当前问题相关"的语义关系。** 关联 ≠ 相关。

### 1.2 为什么跳数限制不够

| 跳数限制 | 问题 |
|---------|------|
| 1 跳 | 过于保守，2 跳的直接相关记忆被斩断 |
| 2 跳 | 大多数场景恰好，但对强关联链（GNN→DIEN→AB测试，三条都相关）不够 |
| 3 跳 | 隧道视野已出现（GRAG: −0.086 nDCG@5） |
| 固定 N | 对简单对话过深，对复杂多跳推理可能不够 |

**结论**：跳数是症状，不是病因。正确的停止条件应基于**语义相关性**而非**图的拓扑距离**。

## 2. 理论基础：四篇关键论文

### 2.1 SIM-RAG — Critic 模式的标准答案（SIGIR 2025）

Yang et al., *"Knowing You Don't Know: Learning When to Continue Search in Multi-round RAG through Self-Practicing"*, SIGIR 2025.

**架构**：
```
Reasoner (LLM) → Retriever → Critic (lightweight model)
    ↑                              │
    └──── 如果 Critic 认为不够 ────┘
                    │
                    └── 够了 → 生成最终回答
```

**Critic 设计**：
- 轻量模型，仅判断"当前检索到的信息是否足够回答问题"
- 通过 Self-Practicing 算法生成合成训练数据（inner monologue + retrieval paths），无需昂贵的人工标注
- 同时防止 Over-Confidence（信息不足就回答）和 Over-Retrieval（信息够了还继续搜）

**对 NaNaGi 的启示**：Critic 不生成内容，只做一个二分类判断——"这条新记忆和当前对话相关吗？"这是最轻量的监督形式。

### 2.2 Lightweight Relevance Grader — 小模型的惊人能力（ICICT 2025）

Jeong, *"Lightweight Relevance Grader in RAG"*, IEEE ICICT 2025.

**核心发现**：
> Llama-3.2-1B 微调后做 binary relevance classification，precision 达到 0.775，接近 Llama-3.1-70B 的 0.834。推理 20-50ms，内存 1-2GB。

这意味着一个 1B 级别的轻量模型就可以有效判断 "这条记忆是否和当前对话相关"。不需要 DeepSeek V4 Pro 级别的推理能力。

### 2.3 Semantic Drift Detection — 免模型的检测方法（Zanbaghi, 2025）

Zanbaghi et al., *"Detecting Sleeper Agents in LLMs via Semantic Drift Analysis"*, arXiv:2511.15992.

**方法**：
```
1. 用 Sentence-BERT 将原始用户问题转为 embedding (baseline)
2. 每检索一条新记忆，将其 description 转为 embedding
3. 计算 cosine similarity(new_embedding, baseline_embedding)
4. 如果 similarity < baseline_mean − 3σ → z-score < −3 → semantic drift detected
```

**效果**：92.5% 准确率，100% 精确率（零误报），<1s 每次查询。

**对 NaNaGi 的适用性**：不需要训练。Sentence-BERT 是现成的。检测逻辑是纯计算。这是最轻量的第一道防线。

### 2.4 GRAG — 隧道视野的定量证据（2026）

*"Goal-Relative Adaptive Graph Retrieval"*, Zenodo 2026。

**关键数据**：
> Goal-directed graph navigation improves disambiguation (+0.202 nDCG@5) but creates "retrieval tunnel vision" on multi-hop queries (−0.086 nDCG@5).

这直接验证了你的直觉：图遍历确实会导致隧道视野，在 multi-hop 场景下有统计显著的负效果。

## 3. 方案设计：三层监督体系

### 3.1 总体架构

```
用户消息
  │
  ▼
Stage 1: MEMORY.md 索引 → side-query LLM → top-5 候选
  │
  ▼
Stage 2: 顺序打开候选文件 → 每条命中检查 ## 关联
  │
  ▼
┌─────────────────────────────────────────────────────┐
│              🆕 Supervisor 三层监督                   │
│                                                     │
│  Layer A: Semantic Drift Detector (确定性, 零成本)    │
│    ↓ 通过                                           │
│  Layer B: Lightweight Critic (小模型, 低延迟)        │
│    ↓ 通过                                           │
│  Layer C: Relevance Trajectory (统计趋势)             │
│    ↓ 任一 Layer 拒绝 → 终止关联遍历                    │
└─────────────────────────────────────────────────────┘
  │
  ▼
组装记忆上下文 → system prompt → NaNaGi 回复
```

### 3.2 Layer A: Semantic Drift Detector（第一道防线）

**原理**：Zanbaghi (2025) 的 cosine drift + z-score 方法。

**实现**：
```typescript
// 初始化：用户消息 → embedding
const baseline = await embed(userMessage); // Sentence-BERT, 384d

// 每条关联记忆：description → embedding → cosine similarity
function checkDrift(memoryDesc: string, baseline: number[]): boolean {
  const memEmb = await embed(memoryDesc);
  const cosSim = cosineSimilarity(memEmb, baseline);
  // 阈值：如果相似度 < 0.3，说明该记忆与原始问题几乎无关
  // 参考: Zanbaghi 的 safe μ=0.57, backdoor μ=0.11, 差距清晰
  return cosSim >= 0.3; // true = 通过, false = 漂移
}
```

**特点**：
- 免训练：Sentence-BERT（paraphrase-multilingual-MiniLM-L12-v2）开箱即用
- 零 LLM 调用：纯向量数学
- 延迟：< 10ms（本地 embedding + cosine）
- 作为第一道防线，过滤明显的语义偏离

### 3.3 Layer B: Lightweight Critic（第二道防线）

**原理**：SIM-RAG Critic 的二元判决模式 + Lightweight Relevance Grader 的大小模型能力对齐发现。

**实现**：
```typescript
// 轻量 Critic prompt — 只做二分类，不生成内容
const CRITIC_PROMPT = `你是一个记忆相关性判断器。只回答 YES 或 NO。

## 用户当前问题
${userMessage}

## 候选记忆描述
${memoryDescription}

这条记忆是否与用户的问题直接相关？只回答 YES 或 NO。`;
```

**模型选择**：OpenAI GPT-4o-mini / Claude Haiku / DeepSeek Lite
- 这些模型做"二分类判断"的能力接近旗舰模型
- Lightweight Relevance Grader 的发现：1B 模型的 precision (0.775) 接近 70B (0.834) — 对于二分类任务足够了
- 延迟：200-500ms
- 成本：每次判断 < 100 tokens

**判决规则**：
```
Critic 返回 YES → 该记忆相关 → 读正文 → 继续关联遍历
Critic 返回 NO  → 该记忆不相关 → 终止关联遍历
Critic 连续 2 次 NO → 硬终止（不再尝试当前链路上的任何关联）
```

### 3.4 Layer C: Relevance Trajectory（第三道防线）

**原理**：Wei (2025) 的 representation drift 轨迹分析。不只看当前这一步，而看整个检索链的相关性趋势。

**实现**：
```typescript
interface RelevancePoint {
  step: number;          // 第几步（0 = 直接命中, 1/2/3 = 关联跳数）
  similarity: number;    // cosine similarity with baseline
  criticScore: number;   // Critic 的置信度 (logit)
}

// 隧道视野检测：连续 2 步的 relevance 单调下降 → 触发终止
function detectTunnel(relevanceHistory: RelevancePoint[]): boolean {
  if (relevanceHistory.length < 2) return false;
  const recent3 = relevanceHistory.slice(-3);
  // 检查是否单调递减
  for (let i = 1; i < recent3.length; i++) {
    if (recent3[i].similarity >= recent3[i-1].similarity) return false;
  }
  // 如果最近 3 步的 similarity 都在下降 → 隧道视野
  return true;
}
```

**为什么这是隧道视野的特异性检测**：
- 真正的多跳推理：relevance 可能波动（第二步降，第三步因为有新发现而回升）
- 隧道视野：relevance 单调下降——每一步离原始问题更远
- 这个模式在 GRAG (−0.086 nDCG@5) 和 Wei (JS-Drift ~0.69 threshold) 中都被独立验证

### 3.5 三层协作逻辑

```
对每条 [[关联]] 内存:

1. Layer A (Semantic Drift):
   cosine(baseline, memory_desc) < 0.3? → 🔴 拒绝, 终止
   
2. Layer B (Critic):
   Critic(user_message, memory_desc) == "NO"? → 🔴 拒绝, 终止
   
3. Layer C (Trajectory):
   detectTunnel(relevanceHistory) == true? → 🔴 拒绝, 终止
   
4. 全部通过 → 读正文, 记录 RelevancePoint, 继续遍历

硬终止条件（无论三层结果如何）:
  - 跳数达 4 → 防止无限遍历 (安全网, 不是主要机制)
  - 累计记忆 token > context 30%
  - 同一 slug 重复
```

## 4. 设计评判

### 4.1 三层是否冗余？

| Layer | 检测什么 | 为什么不能省略 |
|-------|---------|--------------|
| A (Drift) | 明显的语义偏离 | 零成本拦截 80% 的隧道视野，不让简单问题消耗 Critic 调用 |
| B (Critic) | 微妙的不相关 | Drift 可能漏掉"用词相似但实际无关"的情况（如"GNN"出现在完全不同的上下文中） |
| C (Trajectory) | 渐变式漂移 | 单条记忆可能通过 A 和 B，但整体趋势在走偏——这是 Wei (2025) 发现的核心模式 |

**反冗余论证**：Layer B 的 Critic 调用不是每条记忆都触发——只有 Layer A 通过后才进 B。若 80% 被 A 拦截，实际 B 调用远少于遍历步数。Layer C 是纯计算（已有 embedding），不增加 API 调用。

### 4.2 为什么不用 full agent 做监督

MAVEN、PROClaim、MARCH 都用了完整 LLM 作为 adversarial verifier——效果很强，但引入：
- 额外 LLM 调用的延迟（消息来回 × N）
- 对话从"人-NaNaGi"变成"人-NaNaGi-Supervisor"三向——用户等待时间翻倍

SIM-RAG 的路子更适合 NaNaGi：Critic 不是 agent，是一个**轻量判断函数**。不参与对话流，不影响用户感知延迟。

### 4.3 与 Claude Code 的对比

| 维度 | Claude Code | NaNaGi 三层监督 |
|------|-------------|---------------|
| 关联发现 | side-query LLM 每次重新判断 | side-query LLM + 持久化 [[关联]] |
| 关联验证 | ❌ 无显式关联，不需要验证 | ✅ 三层监督验证每条关联 |
| 隧道视野 | 不存在（无图遍历） | 三层防御 + 跳数安全网 |
| 额外 LLM 调用 | side-query (Sonnet) 每轮 1 次 | side-query + 可能的 Critic 调用 |
| 可审计性 | 低（side-query 轨迹内部） | 高（每层判断有日志） |

NaNaGi 多做了一层关联遍历（Claude Code 没有的），所以需要额外的隧道视野防御。这不是过度工程——这是对 GRAG 论文发现的 −0.086 nDCG@5 隧道视野风险的针对性缓解。

### 4.4 对 NaNaGi 的具体适配

**worker 面具**：Layer B 的 Critic prompt 需要调整——不是查"是否和对话主题相关"，而是查"是否和搜索目标相关"。论文检索场景下，Drift 阈值需要收紧（学术相关性的边界更窄）。

**知识库规模增长**：当 per-user 记忆从 50 条增长到 500 条，side-query 的 manifest 扫描从 ~1KB 变为 ~10KB——仍在可控范围。但 Critic 的 Layer A 拦截比例会升高（更多无关记忆需要排除），这恰好验证了 Drift 前置的价值。

**成本模型**：
```
每次关联遍历的最坏 LLM 调用:
  - side-query: 1 次 (已计划)
  - Critic 调用: ≤ 关联跳数 × (1 − Layer A 拦截率)
    - 假设 4 跳, 80% 拦截率 → ~0.8 次 Critic 调用
  - 总计: ~1.8 次轻量 LLM 调用/轮对话
```
可接受。且 Critic 可批量——多条关联描述一次发送，一次调用判断多条。

## 5. 实现建议

### 5.1 分阶段实施

| 阶段 | 内容 | 依赖 |
|------|------|------|
| P0 (当前) | Layer A (Semantic Drift) + 跳数安全网 | Sentence-BERT npm 包, embedding 库 |
| P1 | Layer B (Lightweight Critic) | DeepSeek API（已有） |
| P2 | Layer C (Trajectory) | 依赖 P0 的 embedding 基础设施 |

P0 即可解决 80% 的隧道视野。P1 补上语义微妙差异的检测。P2 是锦上添花。

### 5.2 文件改动

| 文件 | 改动 |
|------|------|
| `lib/embedding.ts` | 🆕 Sentence-BERT embedding 封装 |
| `lib/memory-retrieval.ts` | 🆕 检索管线：side-query → Drift → Critic → Trajectory |
| `agent/tools/search-memory.ts` | 接入新检索管线 |
