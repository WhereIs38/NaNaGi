## 三、问题汇总

| # | 严重度 | 谁 | 问题 |
|---|--------|-----|------|
| 1 | 🔴 | Admin | 框架拦截记忆写入 `data/admin/memories/`，从 `data/memory/` 读取 — **写了白写** |
| 2 | 🔴🔴 | Guest | `search_memory` 调 `lib/memory.ts`，无 personId 过滤 — **隐私泄露**，guest 搜到 admin 记忆 |
| 3 | 🟡 | Admin | 两条写路径落不同目录 — `save_memory` 工具 → `data/memory/`，框架拦截 → `data/admin/memories/` |
| 4 | 🟡 | Admin | `buildMemoryContext` 读 `data/memory/` 全量 — admin 和所有 guest 记忆混在一起 |

## 四、修复方案（最小化）

**原则**：不改 Guest 的 LevelDB 闭环（它自身读写一致），只修两个事：

### 修 1：统一 Admin 的记忆落点

让 `store.ts` `createMemory` 对 admin 走 `lib/memory.ts`，不再写 `data/admin/memories/`。

```
store.ts:79-85  →  改为调用 lib/memory.ts createMemory()
```

### 修 2：`lib/memory.ts` 按 personId 分目录

```
data/memory/
├── nanzhijin/           ← admin 专属
│   ├── MEMORY.md
│   └── mem-*.md
├── guest-4gZd7stK9eQZ/  ← guest (未来迁移，当前不动)
│   └── ...
```

### 修 3：`search_memory` 加 personId 过滤

```
search-memory.ts:37  →  listMemories(ctx.personId)
```

### 修后效果

```
Admin:
  写: save_memory 工具 + 框架拦截 → 都走 lib/memory.ts → data/memory/nanzhijin/  ✅
  读: buildMemoryContext + search_memory → data/memory/nanzhijin/  ✅
  
Guest:
  写: save_memory + 框架拦截 → LevelDB (不变)  ✅
  读: buildMemoryContext → LevelDB (不变)  ✅
      search_memory → data/memory/{personId}/ (隔离)  ✅
```
