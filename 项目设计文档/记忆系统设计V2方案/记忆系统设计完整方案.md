## A1. 设计概述

```
用户消息
  │
  ▼
Stage 1: side-query LLM 扫 MEMORY.md manifest → top-5 相关记忆
  │
  ▼
Stage 2: 顺序打开命中文件，读正文
  → 检查底部 ## 关联 中的 [[links]]
  │
  ▼
Stage 3: Supervisor 三层监督（Part B 详述）
  → 每条关联记忆过 Layer A (Drift) → Layer B (Critic) → Layer C (Trajectory)
  → 任一拒绝 → 终止
  → 全部通过 → 追加读取
  │
  ▼
Stage 4: 硬终止安全网
  → 跳数达 4 / 累计 token > context 30% / 去重命中
  │
  ▼
Stage 5: 组装记忆上下文 → system prompt → NaNaGi 回复
```

## A2. Layer 1: 索引先行 — MEMORY.md 是否必要？

### 正面证据

**Claude Code 证明了索引模式的生产可行性。** Claude Code 的记忆系统在每轮对话开始时读取 `MEMORY.md` 索引文件，然后通过 side-query 调用 Sonnet 做 relevance ranking。这个架构支撑了数十万开发者的日常使用。

**Context Cartography (2026) 提供了理论基础。** 该论文提出了"三元空间模型"——Black Fog（不可见）、Gray Fog（已存储但未激活）、Visible Field（活跃推理面）。MEMORY.md 索引的角色正是将 Gray Fog 中的记忆高效映射到 Visible Field：

> "The core challenge of context engineering is not storage but *selection* — determining which subset of the Gray Fog enters the Visible Field for a given turn." (Context Cartography, 2026)

索引文件 = Gray Fog → Visible Field 的 gate。没有索引，每次都要遍历全目录——复杂度 O(n) 且每多一个文件就多一次 `readFile`。

**Memory for Autonomous LLM Agents (Du, 2026)** 将 agent 记忆形式化为 write–manage–read 循环内的 POMDP。索引是"manage"阶段的核心数据结构——它决定了 read 阶段的效率。

### 反面论证

**索引文件存在单点故障。** Claude Code 的 MEMORY.md 有 200 行/25KB 硬上限，超限**静默截断**——底部的记忆指针消失，对应文件变成孤儿（orphaned files），Claude 不知道它们丢了（GitHub Issue #40614, anthropics/claude-code）。NaNaGi 必须处理这个边界条件。

**简单关键词匹配不如 embedding 检索。** Mem0、claude-cortex、continuity-v2 等社区项目已用向量数据库替代 flat markdown 索引。但向量数据库引入额外依赖（LanceDB/Chroma），P5-B2 阶段再评估。

### 判断

**MEMORY.md 索引是正确的选择，但需要硬上限保护。**

| 条件 | 索引方案 | 无索引遍历 |
|------|---------|-----------|
| 10 条记忆 | 1 次 readFile (索引) + N 次 readFile (命中) | 10 次 readFile |
| 100 条记忆 | 1 次 readFile + ~5 次 readFile | 100 次 readFile |
| 索引膨胀到 200 行 | 截断保护 → 旧记忆进入"仅搜索"模式 | 遍历越来越慢 |

索引上限 150 行。超限时分组折叠（按 type 合并为一行 `## 👤 访客档案 (12条)`），保持索引在可控范围。

## A3. Layer 2: 文件名策略 — slug vs 自然语言

### 正面证据

**文件名即摘要可消除元数据检索层。** 传统方案：文件名是 UUID → 需要打开文件读 frontmatter → 匹配 description。自然语言文件名：`ls` 一下就知道内容，省一次 I/O。

### 反面论证

**中文文件名在跨平台文件系统上不可靠。** Windows 的 NTFS 支持中文文件名，但 git 在不同 OS 间传输时可能出现编码问题。文件名长度有文件系统限制（≤255 字节）。

**EcphoryRAG (Liao, 2025)** 给了一个折中方案：保留 kebab-case slug 作为文件系统标识符，但让索引文件承担语义检索的全部职责——文件名不需要被"读懂"，索引行才需要。

### 判断

**双标识策略：kebab-case 文件名 + 中文 description 在索引中。**

```
文件名: mianshiguan-zijietiaodong-tuijian.md
MEMORY.md 索引行: - [面试官来自字节跳动，关注推荐系统](mianshiguan-zijietiaodong-tuijian.md) — 2026-06-07
```

理由：文件名跨平台安全（纯 ASCII）；索引行承担语义检索（NaNaGi 扫 MEMORY.md 而不是 `ls`）；`description` 字段在 frontmatter 中保留，作为索引行的生成源；与 Claude Code 命名惯例一致。

## A4. Layer 3: [[关联]] 图遍历 — 多跳检索的价值与风险

### 正面证据

**SAGE (Wang et al., 2026)** 证明了图结构记忆的自演化价值。其核心机制是 memory writer 从交互中增量构建异构图，memory reader 做多跳检索：

> "After 2 self-evolution rounds, SAGE achieves best average rank on multi-hop QA; improves long-term memory & hallucination metrics on LongMemEval and HaluMem." (SAGE, 2026)

[[关联]] 扮演的角色与 SAGE 的图边相同——显式编码记忆之间的语义关系，使检索从"找到一条"变为"找到一片"。

**EcphoryRAG (Liao, 2025)** 从认知心理学角度支持关联检索。它借鉴人类"线索回忆"（cued recall / ecphory）机制——一个记忆线索激活与之关联的其他记忆。这恰是 `## 关联` 段落的设计意图：

> "EcphoryRAG extracts 'cue entities' from queries, then performs multi-hop associative search over a lightweight entity graph." (EcphoryRAG, 2025)

**HiGraAgent (Luu et al., 2026)** 的双 Agent 协议（Seeker + Librarian）表明，即便在没有显式图结构的情况下，通过层次化 KG + 混合检索也能实现 +11.7% 的准确率提升。NaNaGi 的 [[关联]] 可视为最轻量级的图边实现。

### 反面论证

**[[关联]] 是手动维护的，不可靠。** Claude Code 的记忆系统没有显式图遍历机制。它的 [[link]] 语法只是 markdown 导航辅助，而非检索管线的一部分。真正的关联发现由 side-query relevance ranking 完成。

NaNaGi 的 [[关联]] 由 `save_memory` 时 LLM 自动生成——扫描已有记忆，判断新记忆是否与已有记忆相关，写入 [[links]]。这需要额外一次 LLM 调用，但可以放在后台执行。

**GRAG (2026) 的"检索隧道视野"风险。** GRAG 在 goal-directed 检索中发现了负效果：

> "Goal-directed graph navigation improves disambiguation (+0.202 nDCG@5) but creates 'retrieval tunnel vision' on multi-hop queries (−0.086 nDCG@5), following graph edges away from the user's actual intent." (GRAG, 2026)

如果 NaNaGi 沿着 [[关联]] 链路越走越远，可能被引导到与当前对话无关的记忆。**关联 ≠ 相关。** 这是 Part B 监督机制要解决的核心问题。

**与 Claude Code 的关键对比：**

| 维度 | Claude Code | NaNaGi |
|------|-------------|--------|
| 关联发现 | side-query LLM 每次重新判断 | side-query LLM + 持久化 [[关联]] |
| 关联验证 | ❌ 无显式关联 | ✅ 三层监督验证每条关联 |
| 隧道视野 | 不存在（无图遍历） | 三层防御 + 跳数安全网 |

NaNaGi 多做了一层关联遍历（Claude Code 没有的），所以需要额外的隧道视野防御——这是 Part B 的设计动机。

### 判断

**[[关联]] 保留为主动检索路径，但受监督机制约束。** `## 关联` 由 NaNaGi 在 `save_memory` 时自动生成，读记忆时沿关联链路遍历，每一步过 Supervisor 三层验证。

## A5. Layer 4: 截断规则 — 停止条件的理论基础

### 相关论文

**Stop-RAG (Park et al., NeurIPS 2025)** 将迭代检索建模为有限时域 MDP，训练 Q-network 学习停止策略：

> "Present-focused confidence signals are unreliable — even when all necessary evidence has been retrieved, a large volume of irrelevant documents can yield low confidence scores that falsely trigger continued retrieval." (Stop-RAG, 2025)

不能用"找到多少条"来判断是否该停——因为可能找到的都是不相关的。

**TASR (2026)**：当模型重复上一轮答案且校准后 logit margin > 0.25 时停止。免训练，一个谓词。

> "RLHF models' verbalized confidence is severely degraded (96.5% of values concentrated at 5). Logit margin provides 44× better class-conditional separation." (TASR, 2026)

**IterResearch (ICLR 2026)**：Agent 在 2,048 轮交互中自主学会停止（平均 ~80 轮），仅用 40K 上下文窗口，核心机制是"演化报告"而非累积消息历史。

### NaNaGi 的场景特殊在哪

上述论文处理的都是**单次任务的迭代检索**。NaNaGi 的记忆检索是**对话的前置步骤**——检索结果用于注入 context，不是直接展示给用户。截断过晚 → context 膨胀（论文显示标称 200K 的模型在 50K 处就开始性能下降）；截断过早 → 遗漏关键记忆。

### 截断规则

借鉴 TASR 的"免训练、可审计"哲学和 Focused ReAct 的"重复检测"：

```
停止条件 (满足任一即停):

P0 (安全): 无更多 [[关联]] / 所有关联已读过 / 跳数达 4
P1 (质量): Supervisor 三层任一拒绝 → 终止
P2 (容量): 累计记忆 token 数 > context 预算 × 30%
P3 (循环): 命中已读 slug → 去重跳过 (Focused ReAct 的 early stop)
```

**不做固定计数截断。** 累计记忆 5 个太死板——频繁对话的用户可能有 8 条相关记忆。取代方案：Part B 的 Supervisor 按语义相关性动态裁决，而非按计数硬截。

## B1. 隧道视野问题定义

```
用户问: "GNN-B 的朋友圈权重方案后来怎么样了？"

MEMORY.md + side-query → GNN-B边权重设计.md
  └─ [[关联]] → gnn-dien-temporal-interest.md (GNN+DIEN时序)
       └─ [[关联]] → sparrl-positioning-decision.md (SparRL定位决策)
            └─ [[关联]] → nanagi-psychology-research-device.md (心理学研究装置)
                 └─ 🔴 已完全偏离GNN话题，进入NaNaGi设计哲学

跳数: 4。第3-4条记忆与原始问题无关，但被图边牵引至此。
```

**根因**：图边编码的是"曾经一起讨论过"的共现关系，而非"针对当前问题相关"的语义关系。**关联 ≠ 相关。**

### 为什么跳数限制不够

| 限制方式 | 问题 |
|---------|------|
| 1 跳 | 过于保守，2 跳的直接相关记忆被斩断 |
| 2 跳 | 大多数场景恰好，但对强关联链（GNN→DIEN→AB测试，三条都相关）不够 |
| 3 跳 | 隧道视野已出现（GRAG: −0.086 nDCG@5） |
| 固定 N | 对简单对话过深，对复杂多跳推理可能不够 |

跳数是症状，不是病因。正确的停止条件应基于**语义相关性**而非**图的拓扑距离**。

## B2. 理论基础：四篇关键论文

### SIM-RAG — Critic 模式的标准答案（SIGIR 2025）

Yang et al., *"Knowing You Don't Know: Learning When to Continue Search in Multi-round RAG through Self-Practicing"*, SIGIR 2025.

```
Reasoner (LLM) → Retriever → Critic (lightweight model)
    ↑                              │
    └──── 如果 Critic 认为不够 ────┘
                    │
                    └── 够了 → 生成最终回答
```

Critic 不生成内容，只做二分类判断——"信息够了吗？"。通过 Self-Practicing 算法生成合成训练数据（inner monologue + retrieval paths），无需人工标注。同时防止 Over-Confidence（信息不足就回答）和 Over-Retrieval（信息够了还继续搜）。

### Lightweight Relevance Grader — 小模型的惊人能力（ICICT 2025）

Jeong, *"Lightweight Relevance Grader in RAG"*, IEEE ICICT 2025.

> Llama-3.2-1B 微调后做 binary relevance classification，precision 达到 0.775，接近 Llama-3.1-70B 的 0.834。推理 20-50ms，内存 1-2GB。

一个 1B 级别的轻量模型就可以有效判断"这条记忆是否和当前对话相关"——不需要 DeepSeek V4 Pro 级别的推理能力。

### Semantic Drift Detection — 免模型的检测（Zanbaghi, 2025）

Zanbaghi et al., *"Detecting Sleeper Agents in LLMs via Semantic Drift Analysis"*, arXiv:2511.15992.

```
1. 用 Sentence-BERT 将原始用户问题转为 embedding (baseline)
2. 每检索一条新记忆，将其 description 转为 embedding
3. 计算 cosine similarity(new_embedding, baseline_embedding)
4. similarity < 0.3 → semantic drift detected
```

效果：92.5% 准确率，100% 精确率（零误报），<1s 每次查询。免训练，纯向量数学。

### GRAG — 隧道视野的定量证据（2026）

*"Goal-Relative Adaptive Graph Retrieval"*, Zenodo 2026.

> Goal-directed graph navigation improves disambiguation (+0.202 nDCG@5) but creates "retrieval tunnel vision" on multi-hop queries (−0.086 nDCG@5).

直接验证了图遍历会导致隧道视野，在 multi-hop 场景下有统计显著的负效果。

## B3. 方案设计：三层监督体系

### 总体架构

```
Stage 1: side-query LLM → top-5 候选记忆
Stage 2: 顺序打开，读正文 → 检查 ## 关联
  │
  ▼
┌─────────────────────────────────────────────────────┐
│              🆕 Supervisor 三层监督                   │
│                                                     │
│  Layer A: Semantic Drift Detector (确定性, 零 LLM)   │
│    ↓ 通过                                           │
│  Layer B: Lightweight Critic (小模型, 二分类)        │
│    ↓ 通过                                           │
│  Layer C: Relevance Trajectory (统计趋势)             │
│    ↓ 任一 Layer 拒绝 → 终止关联遍历                    │
└─────────────────────────────────────────────────────┘
  │
  ▼
硬终止安全网: 跳数 4 | token 30% | 去重
  │
  ▼
组装记忆上下文 → system prompt → NaNaGi 回复
```

### Layer A: Semantic Drift Detector

**原理**：Zanbaghi (2025) 的 cosine drift 方法。

```typescript
const baseline = await embed(userMessage); // Sentence-BERT, 384d

function checkDrift(memoryDesc: string, baseline: number[]): boolean {
  const memEmb = await embed(memoryDesc);
  const cosSim = cosineSimilarity(memEmb, baseline);
  // similarity < 0.3 → 该记忆与原始问题几乎无关
  // Zanbaghi: safe μ=0.57, backdoor μ=0.11, 差距清晰
  return cosSim >= 0.3;
}
```

特点：免训练（Sentence-BERT 开箱即用）、零 LLM 调用（纯向量数学）、<10ms 延迟、作为第一道防线过滤 80% 的明显偏离。

### Layer B: Lightweight Critic

**原理**：SIM-RAG Critic 的二元判决 + Lightweight Relevance Grader 的能力对齐发现。

```typescript
const CRITIC_PROMPT = `你是一个记忆相关性判断器。只回答 YES 或 NO。

## 用户当前问题
${userMessage}

## 候选记忆描述
${memoryDescription}

这条记忆是否与用户的问题直接相关？只回答 YES 或 NO。`;
```

模型：DeepSeek Lite / GPT-4o-mini / Claude Haiku。1B 级别的 precision 接近 70B——对于二分类任务足够。

判决规则：
- Critic YES → 该记忆相关 → 读正文 → 继续
- Critic NO → 不相关 → 终止
- 连续 2 次 NO → 硬终止当前链路

### Layer C: Relevance Trajectory

**原理**：Wei (2025) 的 representation drift 轨迹分析。不只看当前这一步，而看整个检索链的相关性趋势。

```typescript
interface RelevancePoint {
  step: number;          // 0 = 直接命中, 1/2/3 = 关联跳数
  similarity: number;    // cosine similarity with baseline
  criticScore: number;   // Critic 的置信度 (logit)
}

function detectTunnel(relevanceHistory: RelevancePoint[]): boolean {
  if (relevanceHistory.length < 3) return false;
  const recent3 = relevanceHistory.slice(-3);
  // 最近 3 步 monotonic 下降 → 隧道视野
  for (let i = 1; i < recent3.length; i++) {
    if (recent3[i].similarity >= recent3[i-1].similarity) return false;
  }
  return true;
}
```

**特异性**：真正的多跳推理 relevance 可能波动（第二步降，第三步因新发现而回升）；隧道视野 relevance 单调下降——每一步离原始问题更远。这个模式在 GRAG (−0.086 nDCG@5) 和 Wei (JS-Drift ~0.69 threshold) 中被独立验证。

### 三层协作逻辑

```
对每条 [[关联]] 内存:

1. Layer A: cosine(baseline, desc) < 0.3? → 🔴 拒绝, 终止
2. Layer B: Critic(user, desc) == "NO"?   → 🔴 拒绝, 终止
3. Layer C: detectTunnel(history) == true? → 🔴 拒绝, 终止
4. 全部通过 → 读正文, 记录 RelevancePoint, 继续遍历

硬终止安全网: 跳数达 4 | token > 30% | 去重命中
```

## B4. 设计评判

### 三层是否冗余？

| Layer | 检测什么 | 为什么不能省略 |
|-------|---------|--------------|
| A (Drift) | 明显的语义偏离 | 零成本拦截 80%，不让简单问题消耗 Critic |
| B (Critic) | 微妙的不相关 | Drift 可能漏掉"用词相似但实际无关"（如"GNN"出现在完全不同上下文） |
| C (Trajectory) | 渐变式漂移 | 单条记忆可能通过 A 和 B，但整体趋势在走偏——Wei (2025) 的核心发现 |

Layer B 不是每条记忆都触发——只有 A 通过后才进 B。80% 被 A 拦截时，实际 B 调用远少于遍历步数。Layer C 纯计算，不增加 API 调用。

### 为什么不用 full agent 做监督

MAVEN、PROClaim、MARCH 都用了完整 LLM 作为 adversarial verifier——效果强，但引入额外 LLM 调用的延迟（消息来回 × N），对话从"人-NaNaGi"变成"人-NaNaGi-Supervisor"三向。SIM-RAG 的路子更适合 NaNaGi：Critic 是轻量判断函数，不参与对话流。

### 成本模型

```
每次关联遍历的最坏 LLM 调用:
  - side-query: 1 次
  - Critic: ≤ 关联跳数 × (1 − Layer A 拦截率)
    - 假设 4 跳, 80% 拦截率 → ~0.8 次 Critic 调用
  - 总计: ~1.8 次轻量 LLM 调用/轮对话
```

Critic 可批量——多条关联描述一次发送，一次调用判断多条。

### worker 面具适配

paper-search / textbook-query 场景下，Layer B 的 Critic prompt 调整为"是否和搜索目标相关"。学术相关性边界更窄，Drift 阈值需收紧。
