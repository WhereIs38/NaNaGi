## 1. 当前设计概述

```
用户消息
  │
  ▼
Step 1: 扫 MEMORY.md 索引 (一次 readFile, ~1-2KB)
  → 关键词匹配 description，命中目标文件
  │
  ▼
Step 2: 打开目标 .md 文件，读正文
  → 检查底部 ## 关联 中的 [[links]]
  │
  ▼
Step 3: 关联图遍历
  → 回到 MEMORY.md 找对应文件
  → 最多 3 跳，累计 ≤ 5 个文件
  → 去重：同一 slug 不重复读
  │
  ▼
Step 4: 截断判断
  → 跳数 ≤ 3 / 累计 ≤ 5 文件 / token ≤ context 30%
  → 满足任一条件 → 停止遍历
  │
  ▼
Step 5: 组装记忆上下文 → 注入 system prompt → NaNaGi 回复
```

## 2. Layer 1: 索引先行 — MEMORY.md 是否必要？

### 2.1 正面证据

**Claude Code 证明了索引模式的生产可行性。** Claude Code 的记忆系统在每轮对话开始时读取 `MEMORY.md` 索引文件（Claude Code memory architecture, Anthropic, 2025-2026），然后通过 side-query 调用 Sonnet 做 relevance ranking。这个架构支撑了数十万开发者的日常使用。

**Context Cartography (2026) 提供了理论基础。** 该论文提出了"三元空间模型"——Black Fog（不可见）、Gray Fog（已存储但未激活）、Visible Field（活跃推理面）。MEMORY.md 索引的角色正是将 Gray Fog 中的记忆高效映射到 Visible Field：

> "The core challenge of context engineering is not storage but *selection* — determining which subset of the Gray Fog enters the Visible Field for a given turn." (Context Cartography, 2026)

索引文件 = Gray Fog → Visible Field 的 gate。没有索引，每次都要遍历全目录——复杂度 O(n) 且每多一个文件就多一次 `readFile`。

**Memory for Autonomous LLM Agents (Du, 2026)** 将 agent 记忆形式化为 write–manage–read 循环内的 POMDP。索引是"manage"阶段的核心数据结构——它决定了 read 阶段的效率。

### 2.2 反面论证

**索引文件存在单点故障。** Claude Code 的 MEMORY.md 有 200 行/25KB 硬上限，超限**静默截断**——底部的记忆指针消失，对应文件变成孤儿（orphaned files），Claude 不知道它们丢了（GitHub Issue #40614, anthropics/claude-code）。

NaNaGi 如果做索引，必须处理这个边界条件：索引文件膨胀后怎么办？

**简单关键词匹配不如 embedding 检索。** Mem0、claude-cortex、continuity-v2 等社区项目已经用向量数据库替代了 flat markdown 索引。Du (2026) 的综述也指出，"retrieval-augmented memory" 正在从关键词匹配向语义检索迁移。但向量数据库引入额外依赖（LanceDB/Chroma），且对 NaNaGi 当前规模（预计每个用户 < 100 条记忆）来说过度工程化。

### 2.3 判断

**MEMORY.md 索引对 NaNaGi 是正确的选择，但需要硬上限保护。** 理由：

| 条件 | 索引方案 | 无索引遍历 |
|------|---------|-----------|
| 10 条记忆 | 1 次 readFile (索引) + N 次 readFile (命中) | 10 次 readFile |
| 100 条记忆 | 1 次 readFile + ~5 次 readFile | 100 次 readFile |
| 索引膨胀到 200 行 | 截断保护 → 旧记忆进入"仅搜索"模式 | 遍历越来越慢 |

**建议**：索引上限 150 行。超限时分组折叠（按 type 合并为一行 `## 👤 访客档案 (12条)`），保持索引在可控范围。

## 3. Layer 2: 文件名语义化 — 自然语言 slug 的优劣

### 3.1 正面证据

**文件名即摘要消除了元数据检索层。** 传统方案：文件名是 UUID → 需要打开文件读 frontmatter → 匹配 description。自然语言文件名：`ls` 一下就知道内容。这在 I/O 层面省了一步。

**Claude Code 的记忆文件命名采用 kebab-case slug**（如 `happy-llm-progress-2026-06-15.md`），而非自然语言文件名。其设计考虑是：
- 文件系统兼容性（中文文件名在 Windows/Linux 间可能乱码）
- URL 安全（记忆 slug 可能出现在 API 路径中）
- Git 友好

### 3.2 反面论证

**中文文件名在跨平台文件系统上不可靠。** Windows 的 NTFS 支持中文文件名，但 git 在不同 OS 间传输时可能出现编码问题。NaNaGi 目前部署在 Windows，但未来可能迁移到 Linux 服务器。

**文件名长度有文件系统限制。** 大多数文件系统限制文件名 ≤ 255 字节。中文 UTF-8 每个字 3 字节，"面试官来自字节跳动关注推荐系统对GNN工业落地方案特别感兴趣" = 81 字节——安全。但更长的 description 会被截断，截断后的文件名语义不完整。

**EcphoryRAG (Liao, 2025)** 给了一个折中方案：保留 kebab-case slug 作为文件系统标识符，但让索引文件承担语义检索的全部职责——文件名不需要被"读懂"，索引行才需要。

### 3.3 判断

**采用双标识策略：kebab-case 文件名 + 中文 description 在索引中。**

```
文件名: mianshiguan-zijietiaodong-tuijian.md
MEMORY.md 索引行: - [面试官来自字节跳动，关注推荐系统](mianshiguan-zijietiaodong-tuijian.md) — 2026-06-07
```

理由：
1. 文件名跨平台安全（纯 ASCII）
2. 索引行承担语义检索（NaNaGi 扫 MEMORY.md 而不是 `ls`）
3. `description` 字段在 frontmatter 中保留，作为索引行的生成源
4. 与 Claude Code 命名惯例一致

**对用户原方案的修正**：不做中文文件名。但索引行的中文 description 完整保留——那才是 NaNaGi 真正"看"到的。

## 4. Layer 3: [[关联]] 图遍历 — 多跳检索的工程价值与风险

### 4.1 正面证据

**SAGE (Wang et al., 2026)** 证明了图结构记忆的自演化价值。其核心机制是 memory writer 从交互中增量构建异构图，memory reader（基于 Graph Foundation Model）做多跳检索。关键发现：

> "After 2 self-evolution rounds, SAGE achieves best average rank on multi-hop QA; improves long-term memory & hallucination metrics on LongMemEval and HaluMem." (SAGE, 2026)

[[关联]] 扮演的角色与 SAGE 的图边相同——显式编码记忆之间的语义关系，使检索从"找到一条"变为"找到一片"。

**EcphoryRAG (Liao, 2025)** 从认知心理学角度支持关联检索。它借鉴人类"线索回忆"（cued recall / ecphory）机制——一个记忆线索激活与之关联的其他记忆。这恰是 `## 关联` 段落的设计意图：

> "EcphoryRAG extracts 'cue entities' from queries, then performs multi-hop associative search over a lightweight entity graph." (EcphoryRAG, 2025)

**HiGraAgent (Luu et al., 2026)** 的双 Agent 协议（Seeker + Librarian）表明，即便在没有显式图结构的情况下，通过层次化 KG + 混合检索也能实现 +11.7% 的准确率提升。NaNaGi 的 [[关联]] 可以视为最轻量级的图边实现。

### 4.2 反面论证——这是最需要批判的部分

**[[关联]] 是手动维护的，不可靠。**

Claude Code 的记忆系统**没有**显式的图遍历机制。它的 [[link]] 语法只是 markdown 导航辅助（给人类看的），而非检索管线的一部分。真正的关联发现由 side-query relevance ranking 完成——每次调用 Sonnet，让它自己判断哪些记忆相关。

NaNaGi 的 [[关联]] 依赖谁维护？如果 NaNaGi 在 `save_memory` 时自动生成关联——她需要扫描 MEMORY.md 判断哪些已有记忆与新记忆相关。这是一个额外的 LLM 调用（类似 Claude Code 的 side-query），成本和延迟都会增加。如果手动维护（用户在记忆文件中手写关联），则必然出现遗漏和过时。

**GRAG (2026) 的"检索隧道视野"风险值得警惕。**

GRAG 在 goal-directed 检索中发现了负效果：过度聚焦关联图导致检索变成"隧道视野"——在多跳场景下，nDCG@5 下降了 0.086，因为 agent 被图的边引导偏离了用户的实际目标：

> "Goal-directed graph navigation improves disambiguation (+0.202 nDCG@5) but creates 'retrieval tunnel vision' on multi-hop queries (−0.086 nDCG@5), following graph edges away from the user's actual intent." (GRAG, 2026)

这意味着：如果 NaNaGi 沿着 [[关联]] 链路越走越远，她可能被引导到与当前对话无关的记忆。**关联 ≠ 相关**。

**多跳检索的 marginal benefit 在 NaNaGi 规模下可能为零。**

现有图记忆系统（SAGE、GeAR、HiGraAgent）的评估基准是 HotpotQA / MuSiQue / 2WikiMultiHopQA——这些是多跳推理问答数据集，需要综合多个文档的信息才能回答。NaNaGi 的记忆检索场景不同：用户问"之前聊过 GNN 吗？"，找到那条记忆就够了。极少需要"从 GNN 记忆跳到 DIEN 记忆再跳到 AB 测试记忆"才能回答一个问题。

**如果 NaNaGi 的记忆规模是每用户 20-50 条，单跳关键词匹配的 recall 已经足够。** 图遍历带来的增量 recall 可能不到 5%，但增加的复杂度和延迟是实实在在的。

### 4.3 判断

**[[关联]] 保留，但降级为"可选增强"而非"检索管线核心"。**

具体策略：

1. **主检索路径 = MEMORY.md 索引扫 + 关键词匹配**（单跳，快，确定性的）
2. **[[关联]] 仅在以下条件触发**：用户明确表达关联意图（"上次聊 GNN 时提到的那个方案是什么？"）→ 打开目标文件 → 检查 `## 关联` → 如果有关联文件且描述匹配当前问题 → 追加读取。**不做自动多跳遍历。**
3. **不做自动图遍历。** 理由：GRAG 的隧道视野风险、边际收益低、NaNaGi 没有 side-query 预算来验证每条边的相关性。
4. **`## 关联` 由 NaNaGi 在 `save_memory` 时自动生成**：调用一次 LLM（可在后台），判断新记忆是否与已有记忆相关，写入 [[links]]。如果成本敏感，可以降级为纯关键词匹配生成关联。

## 5. Layer 4: 截断规则 — 停止条件的理论基础

### 5.1 谁做过这件事

**Stop-RAG (Park et al., NeurIPS 2025)** 将迭代检索建模为有限时域 MDP，训练 Q-network 学习停止策略。核心发现：

> "Present-focused confidence signals are unreliable — even when all necessary evidence has been retrieved, a large volume of irrelevant documents can yield low confidence scores that falsely trigger continued retrieval." (Stop-RAG, 2025)

这意味着**不能用"找到多少条"来判断是否该停**——因为可能找到的都是不相关的。

**TASR (2026)** 提供了一个更简单的方案：当模型**重复上一轮答案**且**校准后的 logit margin > 0.25** 时停止。免训练，一个谓词。关键发现：

> "RLHF models' verbalized confidence is severely degraded (96.5% of values concentrated at 5 out of 1-10 scale). Logit margin provides 44× better class-conditional separation." (TASR, 2026)

**IterResearch (ICLR 2026)** 展示了最激进的方案：Agent 在 2,048 轮交互中自主学会停止（平均 ~80 轮），仅用 40K 上下文窗口。其核心机制是"演化报告"（evolving report）而非累积消息历史。

### 5.2 NaNaGi 的场景特殊在哪

上述论文处理的都是**单次任务的迭代检索**——一个用户问题 → 多次检索 → 综合回答。NaNaGi 的场景不同：

- 记忆检索是**对话的前置步骤**，不是对话本身
- 检索结果用于**注入 context**，不是直接展示给用户
- 截断过晚 → context 膨胀 → 论文显示标称 200K 的模型在 50K 处就开始性能下降
- 截断过早 → 遗漏关键记忆 → NaNaGi "忘了"重要信息

### 5.3 原方案的截断规则分析

| 规则 | 阈值 | 评判 |
|------|------|------|
| 最大跳数 3 | 硬截断 | ✅ 合理。SAGE 和 HiGraAgent 的实验表明 2-3 跳是最优区间 |
| 累计记忆 5 个 | 硬截断 | ⚠️ 偏低。如果用户记忆池有 30 条，5 条只覆盖 17% |
| Token ≤ context 30% | 软截断 | ✅ 合理。Context Cartography 建议记忆占比不超过窗口 25-30% |

**核心问题**：累计记忆 5 个太死板。一个频繁对话的用户可能有 8 条相关记忆。建议改为**按相关性分数截断**而非计数截断。

### 5.4 建议的截断规则

借鉴 TASR 的"免训练、可审计"哲学和 Focused ReAct 的"重复检测"：

```
停止条件 (满足任一即停):

P0 (安全): 无更多 [[关联]] / 所有关联已读过 / 跳数达 3
P1 (质量): 新读文件与前一个文件的 relevance score < 阈值 (关键词重叠 < 2)
P2 (容量): 累计记忆 token 数 > context 预算 × 30%
P3 (循环): 命中已读 slug → 去重跳过 (Focused ReAct 的 early stop 思路)
```

去掉了固定计数 5。改为 P1 的"relevance 递减检测"——如果关联链上的下一个文件与原始问题的关键词重叠已经很低（< 2 个共有词），说明已经偏题（GRAG 的隧道视野），停止遍历。

## 6. Layer 5: frontmatter 最小化 — 字段留存与删除的论证

### 6.1 原方案 vs 用户最终意见

| 字段 | 原方案 (我之前的设计) | 用户最终意见 | 论文/实践依据 |
|------|---------------------|-------------|-------------|
| `name` | ✅ kebab-case slug | ✅ 保留 | Claude Code 标准字段，文件系统安全 |
| `description` | ✅ 一行摘要 | ✅ 保留 | MEMORY.md 索引行的生成源，side-query 的匹配目标 |
| `type` | ✅ user/project/impression/feedback | ✅ 保留 | MEMORY.md 分组标题的来源，Claude Code 的四个 type |
| `createdAt` | ✅ ISO timestamp | ✅ 保留 | 索引排序（recency bias），Memory freshness 计算 |
| `tags` | ✅ 数组 | ❌ 删除 | EcphoryRAG: 关键词检索不需要结构化标签，正文即标签 |
| `metadata` 子对象 | ❌ 不存在 | — | Claude Code memory 文件有此字段（node_type, originSessionId 等内部追踪） |

### 6.2 逐字段论证

**`name` — 必须保留。** 文件系统标识符。中文文件名在 git / API 路径 / 跨平台场景下不可靠。Claude Code、Mem0、Eros Engine 全部使用 ASCII slug。

**`description` — 必须保留。** 这是 MEMORY.md 索引行的生成源。如果删除，索引更新时需要重新打开文件从正文提取摘要。保留它意味着索引更新只需读 frontmatter（不打开正文）。

**`type` — 必须保留。** MEMORY.md 按 type 分组使得 NaNaGi 和人类都能快速定位。去掉 type 后索引变成平铺列表，检索效率下降。Claude Code 严格约束为四个 type，NaNaGi 也采用同样约束。

**`createdAt` — 必须保留。** Context Engineering survey (Mei et al., 2025) 将 recency 列为上下文选择的首要信号。没有时间戳就无法实现 "最近聊过的记忆优先"。

**`tags` — 建议删除。** 理由：
1. NaNaGi 搜的是自然语言（扫 MEMORY.md 索引行），不是遍历标签数组
2. `description` 和正文已包含所有关键词
3. 标签维护负担重——`save_memory` 时需要额外步骤生成标签
4. EcphoryRAG 的主力召回靠的是实体-关键词重叠，不是预定义标签
5. 但保留 `tags` 字段定义在类型中（不写入文件），为未来 P5-B2 的向量检索留接口

### 6.3 最终字段

```yaml
---
name: mianshiguan-zijietiaodong-tuijian
description: 面试官来自字节跳动，关注推荐系统
type: user
createdAt: 2026-06-07T03:44:58.905Z
---
```

四个字段。不多不少。正文 `## 关联` 段落独立于 frontmatter 存在。

## 7. 整体评判与修改建议

### 7.1 正确的部分

1. **MEMORY.md 索引先行** — 正确。有 Claude Code 生产验证 + Context Cartography 理论支撑。
2. **关联联想能力** — 方向正确但执行需要克制。EcphoryRAG 和人类记忆研究都支持线索-关联激活，但 GRAG 的隧道视野风险要求约束遍历深度。
3. **截断机制** — 必要。Stop-RAG、TASR、IterResearch 都表明无截断的迭代检索最终导致性能下降。
4. **格式统一** — 所有用户 `.md` frontmatter，消除 admin `.md` vs guest `.json` 的分裂。

### 7.2 需要修正的部分

| # | 原设计 | 修正 | 依据 |
|---|--------|------|------|
| 1 | 中文文件名 | kebab-case slug，中文描述在索引中 | 文件系统兼容性，Claude Code 惯例 |
| 2 | 自动多跳图遍历 | 单跳关键词匹配为主，关联追加仅在用户意图明确时触发 | GRAG 隧道视野，NaNaGi 规模下边际收益低 |
| 3 | 固定计数截断 (≤5) | 改为 relevance 递减检测 + token 预算 + 去重 | TASR 免训练自适应停止哲学 |
| 4 | `tags` 字段 | 删除 | 描述+正文已覆盖，维护成本 > 检索收益 |
| 5 | `description` 仅做索引行 | `description` 同时承担索引行和 side-query manifest | Claude Code 的 relevance ranking 依赖 manifest |

### 7.3 最终检索流程

```
用户消息
  │
  ▼
Step 1: 扫 MEMORY.md 索引 (~1KB, 一次 readFile)
  → 按 type 分组，按 createdAt 倒序
  → 关键词匹配 description → 命中候选文件列表
  │
  ▼
Step 2: 顺序打开候选文件 (top-3 by recency)
  → 读正文 → 检查 relevance (关键词重叠数)
  → relevance < 阈值 → 跳过
  │
  ▼
Step 3: 仅当用户消息含关联意图时
  → 检查已读文件的 ## 关联
  → 有关联 link → 回 MEMORY.md 找 → 读
  → 最多追加 2 个关联文件
  │
  ▼
Step 4: 截断
  → P0: 无更多候选 / 跳数达 2 (关联遍历)
  → P1: relevance 递减 (新文件与查询重叠词 < 2)
  → P2: 累计 token > context 30%
  → P3: 去重命中
  │
  ▼
Step 5: 组装记忆上下文 → 注入 system prompt
```

### 7.4 与 Claude Code 的关键差异

| 维度 | Claude Code | NaNaGi |
|------|-------------|--------|
| Relevance ranking | LLM side-query (Sonnet) | 关键词匹配 (确定性，零成本) |
| 关联发现 | LLM 自主判断 | [[links]] + 用户意图触发 |
| 索引膨胀 | 200 行硬截断 (静默) | 150 行 + 分组折叠 |
| 多用户隔离 | 单用户系统 | `data/{personId}/memories/` 物理隔离 |
| 记忆格式 | frontmatter .md | 完全相同 |

NaNaGi 不做 side-query 的理由：成本（每次对话多一次 LLM 调用）+ 当前规模不需要（< 100 条/用户，关键词匹配足够）。P5-B2 引入向量检索后可以重新评估。
