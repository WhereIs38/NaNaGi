# NaNaGi 记忆系统分级架构方案

> 2026-06-16
> 问题驱动：MEMORY.md 扁平索引在规模化时不可靠（Claude Code 200行静默截断，MemForge/LongMemEval 验证）→ 需要分级架构 + 真正的 RAG 检索循环 + 上下文压缩
> 参考：Claude Code compaction、Mem0 混合检索、Letta 文件系统、MemOS 预测调度、MEMTIER 三级架构、EpochDB 分级 HNSW

---

## 1. 问题诊断：当前方案的三重瓶颈

### 1.1 MEMORY.md 扁平索引 = 规模化天花板

Claude Code 的 MEMORY.md 有 **200 行/25KB 硬上限**，超限**静默截断**——底部记忆指针消失，对应文件变成孤儿（GitHub Issue #40614）。NaNaGi 如果也依赖单一 MEMORY.md，会面临同样问题：

- 50 条记忆：索引 ~1KB，扫一遍轻量
- 200 条记忆：索引 ~4KB，开始拖慢
- 500 条记忆：索引 ~10KB + frontmatter 全量读取 → 关键词匹配退化
- 知识库场景（论文/GitHub/文档）：数千条 → 扁平索引完全不可用

**结论**：MEMORY.md 只适合作为"热记忆"索引，需要分级架构将温/冷记忆卸载到独立检索系统。

### 1.2 [[关联]] 图遍历 ≠ 真正的检索循环

当前 [[关联]] 做的：读文件 → 发现 link → 跟 link 再读文件。这是**图遍历**，不是检索循环。

真正的 RAG 检索循环（SIM-RAG, SIGIR 2025；IterResearch, ICLR 2026）：

```
Query → Retrieve → Evaluate sufficiency → 
  if insufficient: Reformulate query → Retrieve again →
  if sufficient: Generate answer
```

区别在于 **Evaluate** 和 **Reformulate** 两步——判断检索到的信息是否足够，不够则改写查询重新检索。[[关联]] 只做了"跟随"这一步，没有评估和改写。

### 1.3 缺少上下文压缩 = 长对话记忆污染

NaNaGi 当前没有对话历史的压缩/摘要机制。随着对话增长：
- Context 膨胀 → 标称 200K 的模型在 50K 处就性能下降（论文验证）
- 旧对话中的记忆碎片占据 context，挤占新检索空间
- Claude Code 的 AutoCompact 机制（Snip → Microcompact → Context Collapse → AutoCompact）是成熟方案

## 2. 产业标准：五大平台的记忆架构

### 2.1 Claude Code — 四层压缩 + side-query

```
上下文管理:
  Snip (零成本) → Microcompact (缓存感知) → 
  Context Collapse (虚拟视图) → AutoCompact (完整摘要, 9段式)

记忆检索:
  MEMORY.md 索引 → side-query Sonnet → top-5 → 注入 context
  200行截断 → 旧记忆变为"仅搜索"
```

### 2.2 Mem0 — 混合向量+图谱

```
对话 → 实体/关系提取 →
  ├── 向量嵌入 → 语义相似搜索
  └── 关系三元组 → 图数据库 → 关系追踪
→ 混合检索 → 记忆增/改/删/不操作 自动决策
```

LoCoMo 基准：基础版 66.9% → 图增强版 68.4%。延迟比全上下文方案降低 91%。

### 2.3 Letta — 文件系统 + 工具驱动

```
Core Memory (永不换出) → Recall Storage (FIFO 溢出备份) → Archival Storage (向量检索按需加载)
Agent 通过 grep/search_files/open/close 等文件工具自主检索
```

LoCoMo 基准：GPT-4o mini + 简单文件系统工具 → **74.0%**，超越 Mem0 图方案。

核心发现：**Agent 使用工具的能力 > 底层数据结构的选择。**

### 2.4 MemOS — 预测性三级调度

```
应用接口层 → 记忆调度与管理层（预测模型所需记忆，预加载）→ 
  存储与基础设施层（明文记忆 + 激活记忆 + 参数记忆）
```

核心创新：不等 Agent 查询，系统提前预测并预加载相关记忆。

### 2.5 EpochDB — 分级 HNSW（2026.6 最新）

```
Working Memory (HNSW, RAM, 0.2-0.4ms) → 
  Historical Archive (Parquet, Disk, ~4ms)
```

基准全满分（LoCoMo/ConvoMem/LongMemEval/NIAH），token 节省 55-79%。

## 3. 新方案：NaNaGi 三级记忆架构

### 3.1 总体架构

```
┌─────────────────────────────────────────────────────────────┐
│                    NaNaGi 三级记忆系统                         │
│                                                             │
│  L1: 热记忆 (Hot) — 始终在 context                            │
│  ├── 当前对话的 IWM Node 摘要 (~500 tok)                      │
│  ├── 最近 N 轮对话摘要 (滚动窗口, ~1500 tok)                    │
│  ├── 当前 session 新产生的记忆 (~1000 tok)                     │
│  └── 人格/角色/PersonaMask 定义 (静态, ~1000 tok)              │
│                                                             │
│  L2: 温记忆 (Warm) — MEMORY.md 索引 + 按需检索                │
│  ├── MEMORY.md 索引 (< 150行, ~3KB)                           │
│  ├── Side-query LLM: 用户消息 → manifest score → top-5       │
│  ├── 命中文件 .md: frontmatter + 正文 + [[关联]]              │
│  ├── 语义检索: Sentence-BERT embedding → cosine top-k        │
│  └── [[关联]] 图遍历 (受三层 Supervisor 约束)                   │
│                                                             │
│  L3: 冷记忆 (Cold) — 向量数据库 + 异步归档                     │
│  ├── LanceDB 向量索引 (全量记忆, embedding)                    │
│  ├── 仅当 L2 召回不足时触发 → 扩大检索范围                      │
│  ├── 记忆过期/衰减: Ebbinghaus 遗忘曲线                         │
│  └── 归档压缩: 超 90 天记忆 → 摘要化 → 原始正文冷存储            │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 三级详解

| 维度 | L1 热记忆 | L2 温记忆 | L3 冷记忆 |
|------|----------|----------|----------|
| **存储** | System prompt 内嵌 | `data/{id}/memories/*.md` + MEMORY.md | LanceDB 向量索引 |
| **容量** | ~4K tokens (预算内) | ~150 条记忆 × ~500 tok = ~75K | 无上限 |
| **检索** | 直接注入，无需检索 | MEMORY.md 索引 + side-query LLM + keyword + [[关联]] | Embedding → cosine top-k |
| **延迟** | 0 | < 500ms（side-query + 文件读取） | < 200ms（向量检索） |
| **维护** | 每次对话自动更新 IWM + 滚动摘要 | save_memory 写入 + 自动生成 [[关联]] | 定期后台任务：embedding、衰减、归档 |
| **触发条件** | 每次对话 | 每次对话（side-query）+ 用户意图（[[关联]] 深度遍历） | L2 召回 < 3 条 / 用户明确要求 / 论文检索 |

### 3.3 检索流程图

```
用户消息
  │
  ▼
L1: IWM + 滚动摘要 + 当前 session → 直接注入 (0延迟)
  │
  ▼
L2: side-query LLM 扫 MEMORY.md manifest → top-5 候选
  ├── 关键词命中 < 3? → 降级到 L3 向量检索补充
  ├── 命中 ≥ 3 → 顺序读 .md 文件 → Supervisor 三层验证
  │   └── [[关联]] 遍历 (受 Supervisor 约束)
  │
  ▼
L3 (条件触发):
  ├── 用户消息嵌入 → LanceDB cosine top-20
  ├── 跨 L2 已命中过滤去重
  └── 追加 top-5 进入候选池
  │
  ▼
Supervisor 三层验证 (所有候选):
  Layer A (Drift) → Layer B (Critic) → Layer C (Trajectory)
  │
  ▼
截断: 累计 token > context 30% | 跳数 4 | 去重
  │
  ▼
组装 → system prompt → NaNaGi 回复
```

## 4. 核心组件详设

### 4.1 L1 热记忆：滚动摘要替代上下文堆积

**借鉴**：Claude Code 的 AutoCompact 9 段式摘要 + IterResearch 的"演化报告"

**实现**：
- 每 6 轮对话，LLM 生成 2 句摘要
- 超 12 轮时的旧消息 → **不删**，但仅保留摘要 + 关键引用
- Cell 结束时生成完整 Cell 摘要 → 存入 LevelDB `conv:{personId}:{cellId}` → 新对话 L1 注入 `lastSummary`
- IWM Node 摘要实时更新（6 trait + 最近关键事件）

**Token 预算**：
```
L1 总预算: ~4K tokens
├── 角色注入 (PersonaMask):           500 tok (固定)
├── IWM 关系摘要:                    1500 tok (动态)
├── 滚动对话摘要 (最近 N 轮):         1000 tok
├── 当前 session 新记忆:             1000 tok (缓冲)
└── 环境感知 + 工具准则:              余量
```

### 4.2 L2 温记忆：MEMORY.md 索引 + 双路检索 + side-query

**MEMORY.md 索引**（保留但加保护）：
```
150 行上限。超限时：
  - 按 type 分组折叠: "## 👤 访客档案 (23条)" → 点开查看全部
  - 旧记忆降级到 L3（仅向量检索，不出现在索引中）
  - 记忆 freshness 超过 30 天 → 自动降级到 L3
```

**双路检索**（借鉴 MemForge 的 hybrid，Mem0 的 vector+graph）：
```
路 1 (确定性的): side-query LLM 扫 MEMORY.md manifest → top-5 slug
路 2 (语义的): 用户消息 embedding → cosine(top-k, L2 embedding cache)
融合: RRF (Reciprocal Rank Fusion), 语义权重 1.5×
```

**side-query 设计**（借鉴 Claude Code）：
```
轻量 LLM (DeepSeek Lite / Haiku) 调用:
  Prompt: 用户消息 + MEMORY.md manifest (name + description + type + createdAt)
  输出: top-5 slug (按相关性排序)
  成本: ~200 input tokens + ~50 output tokens
  延迟: ~300ms
```

### 4.3 L3 冷记忆：向量数据库

**选型：LanceDB**

理由：
- 嵌入式，零运维（与 NaNaGi "一个进程跑所有" 原则一致）
- Rust 内核，Node.js binding
- 支持增量写入和过滤
- ROADMAP_V52 的 P5-B2 已规划

**写入**：
- `save_memory` 时同步写入：文件系统（.md）+ LanceDB（embedding + metadata）
- 后台任务：定期将 L2 超 30 天的记忆标记为 `tier: cold`

**检索**：
- L2 关键词命中 < 3 → 触发 L3 补充检索
- 用户显式请求 "搜索所有历史" → 直接搜索 L3
- 论文/知识库检索 → worker 面具直接搜索 L3

**衰减/遗忘**（借鉴 Ebbinghaus, Mem0 验证 −59% token）：
```
记忆新鲜度 = e^(-age/halflife)
  halflife: 默认 14 天 (可配置)
  freshness < 0.2 → 降级到 L3 冷存储
  freshness < 0.05 → 仅保留摘要，删除原始正文
```

### 4.4 RAG 检索循环 — 真正的 loop

**当前 [[关联]] 的问题**：只有"跟随"，没有"评估"和"改写"。

**真正的检索循环**（SIM-RAG + IterResearch）：

```
Round 1: 用户消息 → side-query top-5 → 读文件 → 
  Supervisor 评估: 检索到的信息是否足以回答？
    ├── 足够 → 组装 context → 回复
    └── 不足 → Reformulate query → Round 2

Round 2: 改写后的 query → side-query 再次选 (可能命中不同文件) →
  or: 扩大检索到 L3 → 评估 →
    ├── 足够 → 组装 → 回复
    ├── 不足 → Round 3 (最多 3 轮）
    └── 3 轮仍不足 → 诚实告知 "我记不太清了"
```

**Reformulate 策略**（借鉴 IterResearch 的 Markov 工作空间）：
```
Round 1 的检索结果作为 context 注入 Round 2 的查询改写:
  "原始问题: GNN-B 的朋友圈权重方案？
   已检索到: [GNN-B边权重设计.md 摘要]
   缺失信息: 实现的代码细节、最终 MRR 结果
   改写查询: GNN-B 边权重 SAGEConv edge_weight 实现 MRR 结果"
```

**与 [[关联]] 的协作**：
- [[关联]] 不作为独立检索路径，而是作为 **检索结果的上下文扩展**
- Round 1 检索到文件 → 读正文 → 检查 [[关联]] → 关联文件加入"候选扩展池"
- Supervisor 验证每条关联 → 通过的追加入 context → 不通过的不进入 Reformulate

## 5. 上下文压缩

### 5.1 借鉴 Claude Code 四层模型

| 层 | Claude Code | NaNaGi 适配 |
|----|-------------|-----------|
| Snip | 移除空结果/被拒绝的工具调用 | 移除被 Supervisor 拒绝的记忆检索 |
| Microcompact | 压缩 tool_result 内容（缓存感知） | 记忆正文 > 500 tok 时截断 + 摘要替代 |
| Context Collapse | 虚拟视图，多轮范围替换为摘要 | Cell 摘要 + 滚动窗口 + IWM 摘要 → L1 |
| AutoCompact | 独立 API 调用生成 9 段式摘要 | Cell end 时异步生成 CellSummary → LevelDB |

### 5.2 NaNaGi 特有：对话中的记忆冲突

对话越长 → 新记忆检索与旧对话历史竞争 context 空间。

**解决**：
- L1 保持固定 token 预算（4K）——旧摘要被压缩，不下沉到 L2
- 每轮对话后的 IWM 摘要更新：不是追加，而是**替换**（IterResearch 的"演化报告"思路）
- 长时间对话（> 20 轮）：强制触发 L1 摘要压缩 → 释放空间给新检索

## 8. 关键决策论证

### Q: 为什么用 LanceDB 而不是 pgvector/Chroma/FAISS？

- LanceDB: 嵌入式、零运维、Node.js binding、与 NaNaGi "单进程"哲学一致
- Chroma: Python-first，Node 支持弱
- FAISS: 无持久化，需额外存储层
- pgvector: 需要 PostgreSQL 进程，过度工程化

### Q: 为什么 side-query + embedding 双路而不是纯 embedding？

MemForge 的 LongMemEval 数据：hybrid (RRF fusion) → 93.2% R@5。纯 embedding → 约 85%。双路融合提供了 ~8% 的召回提升。

### Q: RAG 循环 3 轮上限的依据？

SIM-RAG (SIGIR 2025): 2-3 轮是最优区间。IterResearch (ICLR 2026): 平均 ~80 轮，但那是深度研究任务。对话记忆检索的信息需求远小于深度研究，3 轮是安全上限——超过 3 轮仍检索不到 → 诚实告知比继续搜更好。

### Q: 为什么不用 Letta 的纯文件系统方案？

Letta 的 74% LoCoMo 成绩确实超过了 Mem0 的 68.5%。但 Letta 的设计前提是 Agent 通过工具自主管理文件——需要 Agent 对文件语义有充分理解。NaNaGi 的文件系统记忆（.md files）已经是 Letta 风格的"人类可读存储"，但检索层需要向量辅助——因为 side-query LLM 扫 manifest + embedding 融合的 R@5 远高于纯 `grep`。
