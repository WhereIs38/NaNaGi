# NaNaGi 双人格双路检索 + 读写分离设计

> 2026-06-16
> 论证对象：(1) companion走SIM-RAG / worker走IterResearch 的分裂检索策略
>          (2) 写入走文件系统(Letta式) / 读取走向量辅助(Mem0式) 的混合架构
> 原则：严格论证，不回避矛盾和风险

---

## 1. 双人格分裂检索策略

### 1.1 为什么不能统一？

companion 和 worker 的检索场景在三个维度上根本不同：

| 维度 | companion (小女儿) | worker (大女儿) |
|------|-------------------|-----------------|
| **检索目标** | 对话记忆：上次聊了什么、客人偏好、关系历史 | 知识检索：论文内容、GitHub代码、教材章节 |
| **检索深度** | 浅层：1-3 条相关记忆通常足够回答问题 | 深层：需要综合多源信息，跨文档推理 |
| **用户容忍延迟** | 低：对话中等待 3+ 秒会打断交流节奏 | 高：深度研究任务预期需要时间 |
| **停止条件** | 用户意图明确，查到了就够 | 信息充分性需要自主判断，"感觉够了"不可靠 |
| **失败代价** | 低：可以追问澄清，关系不受损 | 高：遗漏关键论文/数据导致结论错误 |

这些差异不是配置参数能调节的——它们要求不同的检索控制策略。

### 1.2 SIM-RAG 适合 companion 的理由

SIM-RAG (Yang et al., SIGIR 2025) 的设计目标与 companion 的检索场景精确对齐：

**Critic 做"信息充分性"判断**：一个轻量 Critic 在每轮检索后询问"当前检索到的信息是否足以回答用户的问题？"这正是 companion 需要的——对话记忆检索不需要穷举，查到"面试官上次说关注推荐系统"就够了。

**Over-Confidence 防御**：Critic 的存在防止 companion "自以为记得但实际上不够"。这在关系型对话中至关重要——宁可说"让我想想"也比基于不完整记忆给出错误回应好。

**3 轮上限的自然合理性**：
- 对话记忆的关联链通常浅（2-3 跳覆盖绝大多数情况）
- 3 轮内的延迟（每轮 ~500ms side-query + Critic）在对话体验中可接受
- 超过 3 轮意味着检索失败 → 诚实告知比继续搜更符合"小女儿"的人设

### 1.3 IterResearch 适合 worker 的理由

IterResearch (Guoxin et al., ICLR 2026) 的核心机制与 worker 的深度研究场景精确对齐：

**Markov 工作空间替代上下文堆积**：worker 做论文检索时可能遍历数十篇文档。如果每篇都堆积在 context 中 → 50K 处模型性能开始下降。IterResearch 的"演化报告"——一个持续更新的摘要文档取代累积的消息历史——直接解决了这个问题。

**自主停止而非固定上限**：worker 无法预设"查 3 次就够了"。一篇论文引用了另一篇，那篇又引用了奠基性工作——需要多少步取决于研究问题的复杂度。IterResearch 在 2,048 轮交互预算中平均 ~80 轮自主停止——worker 不需要这个数量级，但需要这种自主判断能力。

**BrowseComp 验证的 scaling 特性**：IterResearch 在 BrowseComp 上从 3.5% → 42.5% 的准确率提升，随交互轮次增加而非线性增长——这正是深度研究的 scaling pattern。

### 1.4 分裂的风险

**风险 1：同一记忆池，不同检索行为 → 结果不一致**

同一个用户问同一个问题，小女儿回答"我们聊过这个，你说……"，大女儿回答"根据你之前的讨论和相关文献……"——不同 persona 下检索到不同深度 → 回答不同。

**缓解**：这不是 bug，是 feature。不同面具本就该有不同的表达方式和信息深度。小女儿给的是关系记忆，大女儿给的是知识综合——答案不同是预期行为。

**风险 2：共享 L3（LanceDB）被两个检索策略不同地使用 → 缓存/索引管理复杂**

**缓解**：L3 是纯存储 + 检索层，不感知上层检索策略。两个面具调同一个 `lanceDB.search()` API，差异在调用层（companion 调 1 次 top-5，worker 可能调多次且带 Reformulate）。

**风险 3：Critic 模型在两个场景下的有效性不同**

SIM-RAG 的 Critic 在对话场景有效，但在深度研究场景（需要评估"是否遗漏了关键引用"）可能不够。

**缓解**：worker 不使用 Critic。worker 的停止条件基于 IterResearch 的演化报告收敛（"报告在过去 N 轮未发生实质性变化"），而非 Critic 的二分类判断。

### 1.5 结论

**分裂是正确的。** 两个检索策略共享底层存储（L1/L2/L3），但控制层（检索循环、停止条件、Reformulate 策略）按面具分裂。这不是重复建设——是同一基础设施上的两套控制策略。

```
                    ┌─────────────────────────┐
                    │   L1/L2/L3 共享存储层     │
                    │   (统一格式, 统一索引)    │
                    └──────────┬──────────────┘
                               │
            ┌──────────────────┼──────────────────┐
            │                                     │
   ┌────────┴────────┐                  ┌─────────┴─────────┐
   │ companion 控制层  │                  │  worker 控制层     │
   │                  │                  │                   │
   │ SIM-RAG 循环     │                  │ IterResearch 循环  │
   │ Critic 评估      │                  │ 演化报告收敛       │
   │ 3 轮上限         │                  │ 自主停止 (max 20)  │
   │ 快速 Reformulate │                  │ 深度 Reformulate   │
   └─────────────────┘                  └───────────────────┘
```

## 2. 写入走文件 / 读取走向量 — 读写分离架构

### 2.1 问题重述

你说得对：我们在文件系统上构建记忆（save_memory 工具 → .md + MEMORY.md），这是 Letta 式的——Agent 通过工具自主管理人类可读的记忆文件。但我们不能舍弃向量检索的高效——纯 `grep` 在 500 条记忆中退化严重。

**这个矛盾是假的。写入和读取是两种不同的操作，它们不需要使用同一种技术。**

### 2.2 为什么写入必须走文件系统

**Letta 的核心发现 (LoCoMo 74%) 不是关于"文件系统比向量数据库好"——是关于"Agent 使用工具的能力 > 底层数据结构"。**

```
Letta 的 Agent 做的事:
  save_memory(content) → 打开文件 → 写 .md → 更新索引
  search_memory(query) → 读索引 → grep 关键词 → 读文件

Mem0 的 Agent 做的事:
  add(content) → 后台自动向量化 + 三元组提取
  search(query) → 向量相似度 → 返回结果
```

Letta 胜出的原因不是文件 > 向量。是 Agent 在**显式操作**（"我要打开这个文件，读这段内容"）时利用率更高——因为这种交互模式大量出现在 LLM 的训练数据中，而"我有一个隐式的向量索引在后台"是 LLM 不熟悉的隐喻。

**NaNaGi 已经走上了 Letta 式的写入路径**——`save_memory` 和 `search_memory` 是两个明确的工具，Agent 显式调用它们。保留这个模式意味着保留 Letta 验证过的"工具使用能力 > 数据结构"优势。

### 2.3 为什么读取需要向量辅助

**Mem0 的核心发现 (LoCoMo 66.9% → 68.4% 图增强) 不是关于"向量比文件好"——是关于"语义检索在规模化时不可替代"。**

```
100 条记忆: grep "推荐系统" 命中 8 条, 手工排序 → recall 够用
500 条记忆: grep "推荐系统" 命中 47 条, Agent 无法有效排序 → recall 下降
5000条记忆: 关键词匹配彻底失效, 语义相近但用词不同的记忆完全找不到
```

**MemForge 的 LongMemEval 数据（93.2% R@5 hybrid）验证了双路融合的不可替代性：**

- 纯关键词（FTS + trigram）: R@5 ~78%
- 纯语义（pgvector HNSW）: R@5 ~85%
- 混合（RRF fusion, semantic ×1.5）: R@5 **93.2%**

向量检索不替代关键词——它补充关键词。两者融合的 R@5 比任何单一路径高 8-15 个百分点。

### 2.4 读写分离架构

```
┌─────────────────────────────────────────────────────────┐
│                      写入路径                            │
│                                                         │
│  save_memory 工具 / 框架拦截                             │
│    │                                                    │
│    ├──→ ① 写 .md 文件 (Letta式, 人类可读, 工具可操作)     │
│    │      data/{personId}/memories/{slug}.md             │
│    │                                                    │
│    ├──→ ② 更新 MEMORY.md 索引                           │
│    │      + [description](slug.md) — createdAt           │
│    │                                                    │
│    └──→ ③ 写 LanceDB 向量索引 (异步, best-effort)         │
│           embedding 向量 + {personId, slug, type, desc}  │
│                                                         │
│  ①② = 同步, 强一致 (文件系统先写)                         │
│  ③  = 异步, 最终一致 (写入失败不影响记忆保存成功)           │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                      读取路径                            │
│                                                         │
│  search_memory 工具 / buildMemoryContext                 │
│    │                                                    │
│    ├──→ 路 1: side-query LLM 扫 MEMORY.md manifest      │
│    │         (命中率 ~70%, 低延迟, 零额外基础设施)          │
│    │                                                    │
│    ├──→ 路 2: embedding → LanceDB top-k                 │
│    │         (命中率 ~85%, 补充关键词盲区)                 │
│    │                                                    │
│    └──→ RRF 融合 → top-N → 读文件 (Letta式 显式操作)      │
│                                                         │
│  两路独立运行 → RRF 加权融合 → 最终结果                      │
│  不再做纯 grep (500+ 条记忆时已退化)                       │
└─────────────────────────────────────────────────────────┘
```

### 2.5 为什么读写可以分离 — 严格论证

**论证 1：写入和读取的 bottleneck 不同。**

写入的瓶颈是**语义提取质量**（这条记忆该记什么），不是存储格式。文件系统让 Agent 看到自己在写什么——`save_memory("面试官来自字节跳动")` → Agent 的 tool call 参数本身就是结构化的。而 Mem0 式自动提取可能错误理解对话内容（"他来自字节跳动"可能被误提取为"他在字节跳动工作"，而实际可能是"他面试字节跳动"）。

读取的瓶颈是**规模化召回**——500 条记忆中找出 5 条相关的。向量检索在这个问题上比关键词匹配好 ~15 个百分点（MemForge 数据）。

**写入优化语义提取（Agent 显式操作 > 自动提取），读取优化召回（向量 > 关键词）。两个 bottleneck 的最优解不同 → 读写分离。**

**论证 2：Letta 自己的数据支持这个结论。**

Letta 的 LoCoMo 74% 用的是 GPT-4o mini + 文件系统。但如果把文件系统的检索（grep）换成向量检索，这个数字会更高。Letta 的优势在于"Agent 显式操作文件"这个交互模式，不在于"grep 够好了"。

**论证 3：MemForge 的混合架构是产业验证。**

MemForge 用 PostgreSQL 同时做 FTS（关键词）、pgvector HNSW（语义）、图遍历（关联）——三种检索路径在同一个数据库中，RRF 融合。NaNaGi 的 .md 文件是 MemForge 的 "FTS" 等价物（Agent 可以通过 grep 定位文件），LanceDB 是 "HNSW" 等价物。两者不矛盾——互补。

### 2.6 一致性风险与缓解

**风险：LanceDB 写入失败 → 向量索引与文件不同步 → 检索漏掉新记忆。**

**缓解策略（分层）：**

```
P0 (写入保证): .md 文件 + MEMORY.md 先写 (同步, 强一致)
  → 这一步成功 → 记忆已经被保存 → search_memory 侧路径 1 能查到

P1 (异步补充): LanceDB 写入放在后台队列
  → 成功: 路径 2 也能查到了
  → 失败: 重试 3 次, 指数退避
  → 仍失败: 标记为 pending_vector_sync=true, 下次对话时重试

P2 (一致性修复): 定期后台任务 (MemForge "sleep cycle" 模式)
  → 扫描 .md 文件 → 对比 LanceDB 中的 slug 列表
  → 缺失的 → 重新 embedding → 写入
  → 多余的 (已删除但向量未清理) → 从 LanceDB 删除
```

**风险：文件被删除/更新 → 向量索引残留旧数据。**

**缓解**：检索时交叉验证——LanceDB 返回的 slug 在文件系统中确认存在后才读文件。不存在的 slug → 过滤 + 标记清理。

### 2.7 结论

**写入走文件，读取走向量——两者不矛盾，它们解决的是不同问题。**

这不是妥协方案。这是在保留 Letta 验证过的"Agent 显式操作文件"优势（写路径）的同时，用 Mem0/MemForge 验证过的向量检索弥补规模化召回不足（读路径）。产业中最好的混合系统（MemForge 93.2% R@5）做的就是这件事。

## 3. 整合：两个决策的交叉矩阵

```
                   companion (小女儿)              worker (大女儿)
                   ────────────────               ──────────────
写入路径           文件系统 (.md + MEMORY.md)      同一文件系统
                   Agent 显式 save_memory 工具    同一写入流程

读取路径           双路(LLM manifest + LanceDB)    双路 + 深度检索
                   RRF 融合, top-5                RRF 融合, 无上限

检索循环           SIM-RAG                        IterResearch
                   Critic 评估充分性               演化报告收敛判断
                   3轮上限                         自主停止 (max 20)
                   Reformulate: 轻量改写           Reformulate: 深度改写

停止条件           Critic: "够了" + 轮次上限        报告未变化(3轮) + 
                                                   信息覆盖度达标
```

共享的部分：存储格式（统一 .md frontmatter）、索引结构（MEMORY.md + LanceDB）、写路径（store.ts 统一入口）、Supervisor 三层验证（两个面具都过）。

分裂的部分：检索循环的控制策略——这正是 ENGINEERING_FRAMEWORK 中 PersonaMask 的工程表达：**同一个大脑，不同的思考深度。**
